### Database Module (`database/`)
Built a complete database access layer with four files:

| File | Role |
|------|------|
| `config.py` | Loads `.env`, builds SQLAlchemy engine, connection pool (min 1, max 5), graceful degradation |
| `models.py` | SQLAlchemy ORM definitions for all tables |
| `db_client.py` | Low-level insert/update functions, connection context manager |
| `session_service.py` | Business logic layer between `main.py` and the database |

### Database Schema (initial)
Created four tables:

```
sessions         — one row per pipeline run
passenger_events — one row per passenger crossing
buses            — physical vehicles
stations         — named stops
```

Key design decisions:
- All timestamps stored in UTC with timezone
- `entry_count >= 0` and `exit_count >= 0` CHECK constraints
- Indexes on `session_start`, `timestamp`, `station_id`, `bus_id`
- Parameterized queries throughout (no SQL injection risk)

### Architecture Decision: SessionService Layer
Instead of calling the database directly from `main.py`, introduced a `SessionService` class:

```
main.py → SessionService → db_client → PostgreSQL
```

This means future components (REST API, prediction engine) can reuse the same persistence logic without touching `main.py`.

### CV Integration (`counter.py` + `main.py`)
- Added optional `on_crossing` callback to `LineCounter`
- Callback fires on every confirmed crossing with `(direction, occupancy, timestamp)`
- `main.py` passes `svc.on_crossing` to the counter
- CV layer has zero database imports — fully decoupled

### Session Lifecycle
```
main.py starts
  ↓ SessionService.start()   → inserts session row immediately (so FK is valid)
  ↓ YOLO loop runs
  ↓ each crossing → on_crossing() → inserts PassengerEvent row
  ↓ video ends or user presses q
  ↓ SessionService.end()     → updates session row with final counts
```

## Milestone 2 — Transport Master Data ✅

### Architectural Decision: Master Data vs Transactional Data
Adopted standard BI/data engineering terminology:

| Category | Tables |
|----------|--------|
| **Master Data** (static) | `lines`, `drivers`, `buses`, `stations` |
| **Transactional Data** (append-only) | `sessions`, `passenger_events` |

This separation is standard in enterprise BI and data warehouse design.

### New Tables Added
```sql
lines    — line_id, line_name, line_number
drivers  — driver_id, name, license_number
```

### Altered Tables
```
buses            + license_plate, driver_id (FK→drivers), line_id (FK→lines, now real FK)
sessions         + bus_id (FK→buses)
passenger_events + occupancy_rate NUMERIC(5,2)
```

### Final Relational Chain
```
lines (1) → buses (N)
drivers (1) → buses (N)
buses (1) → sessions (N)
sessions (1) → passenger_events (N)
passenger_events (N) → stations (1)
```

### Key Design Decision: occupancy_rate stored at insert time
**Formula:** `occupancy_rate = (occupancy_after_event / bus.capacity) × 100`

This is **denormalization for analytics performance** — a deliberate trade-off:
- Power BI queries are instant (no runtime joins)
- Bus capacity is cached once at session start, never queried per event
- Acceptable because analytics speed > strict normalization here

Example KPI enabled:
| Bus | Capacity | Occupancy | Rate |
|-----|----------|-----------|------|
| TU-101-AB | 50 | 42 | 84% |
| TU-102-CD | 35 | 18 | 51% |

### Future Architecture Locked In
Created placeholder modules for future milestones:

```
prediction/
    crowd_prediction.py   — "What will happen?" (Milestone 3+)
recommendation/
    dispatcher.py         — "What should we do?" (Milestone 4+)
```

These two modules solve different problems:

Keeping them separate means they can be trained, tested, and replaced independently.
