## Milestone 3 — ETL Pipeline ✅

### Analytics Schema Design

Designed and created **6 new analytics tables** across 3 migration scripts:

#### `migrate_m3_analytics.sql`

| Table | Grain | Purpose |
|-------|-------|---------|
| `hourly_station_statistics` | per station × line × hour | How busy is each station every hour? |
| `line_statistics` | per line × day | Compare transport line performance |
| `bus_statistics` | per bus × day | Monitor every vehicle |
| `etl_watermark` | per pipeline name | Track last successful ETL run |

#### `migrate_m3_daily_system.sql`

| Table | Grain | Purpose |
|-------|-------|---------|
| `daily_system_statistics` | per day (network-wide) | Executive dashboard: total passengers, peak hour, busiest line |

#### `migrate_station_daily.sql`

| Table | Grain | Purpose |
|-------|-------|---------|
| `station_daily_statistics` | per station × line × day | BI-ready station performance with line-level split |

Key design decision: `station_daily_statistics` uses a **three-column unique key** `(date, station_id, line_id)` — not just `(date, station_id)`. This enables Power BI to answer:
- "Which line is most crowded at Bab Saadoun?"
- "Bab Saadoun on Line 23 vs Line 42"

Without `line_id` in the grain, those questions would be unanswerable.

### ETL Module (`etl/`)

Built four Python files:

| File | Role |
|------|------|
| `extract.py` | Reads new `passenger_events` + `sessions` since last watermark, loads master `buses` and `lines` |
| `transform.py` | Validates, cleans, enriches, and aggregates into 5 DataFrames |
| `load.py` | Upserts all 5 analytics tables, updates ETL watermark |
| `run_etl.py` | Pipeline entry point with logging, quality gate, step-by-step reporting |

#### Transform Pipeline (`transform.py`)

Five aggregation functions:

```
clean_events()              → drop invalid rows, flag quality halt if >10% bad
enrich_with_line()          → join line_id from buses master data onto every event
transform_hourly_station()  → group by (hour, station_id, line_id)
transform_line_statistics() → group by (date, line_id)
transform_bus_statistics()  → group by (date, bus_id)
transform_station_daily()   → group by (date, station_id, line_id)
transform_daily_system()    → group by (date) — whole network
```

All aggregations produce: `total_boardings`, `total_alightings`, `avg_occupancy_rate`, `peak_hour`.

Peak hour logic: hour with the **highest mean occupancy rate** for that group that day.

#### Data Quality Gate

In `clean_events()`:
- Drops rows where `occupancy_after_event < 0`, `direction` not IN/OUT, or `timestamp` is null
- If more than **10% of rows are invalid** → sets `quality_halt = True`
- `load_all()` checks this flag and **aborts the load** before writing anything

This prevents corrupt data silently entering the analytics tables.

#### Load Strategy: Upsert (IDEMPOTENT)

All five loaders use `INSERT ... ON CONFLICT DO UPDATE`. This means:
- Running the ETL twice on the same data → same result (no duplicates)
- Reprocessing historical data is safe
- Natural keys per table:

| Table | Natural Key |
|-------|-------------|
| `hourly_station_statistics` | `(hour_start, station_id, line_id)` |
| `line_statistics` | `(date, line_id)` |
| `bus_statistics` | `(date, bus_id)` |
| `station_daily_statistics` | `(date, station_id, line_id)` |
| `daily_system_statistics` | `(date)` |

#### ETL Runner (`run_etl.py`)

- Structured 3-step execution: Extract → Transform → Load
- Logs to both console and `etl/logs/etl_YYYY-MM-DD.log`
- Exits with code `0` on success, `1` on any failure
- Logs row counts per table at each step
- Reports total duration and total rows loaded on completion

### Bug Fixes — Syntax Errors

During code review, discovered **3 syntax errors** where the opening `"""` of a docstring was accidentally merged onto the same `def` line instead of being on a new line. Python raises `SyntaxError` or `IndentationError` at runtime when these functions are called.

| File | Function | Error |
|------|----------|-------|
| `etl/transform.py` | `transform_daily_system` | `SyntaxError` — docstring on def line |
| `etl/load.py` | `load_bus_statistics` | `IndentationError` — body unreachable |
| `etl/load.py` | `update_watermark` | `SyntaxError` — docstring on def line |

All three fixed. Both modules now import cleanly:
```
python -c "import etl.transform; import etl.load; print('OK')"
→ OK
```

## Schema Review — Power BI Readiness ✅

Verified the analytics model supports all three required BI query patterns:

| Query | Supported By |
|-------|-------------|
| Station view: Bab Saadoun total traffic | `station_daily_statistics GROUP BY station_id` |
| Line view: Line 23 performance | `line_statistics WHERE line_id = 23` |
| Combined: Bab Saadoun + Line 23 | `station_daily_statistics WHERE station_id = X AND line_id = 23` |

The `UNIQUE (date, station_id, line_id)` constraint on `station_daily_statistics` guarantees no duplicate rows, confirmed by:
```sql
SELECT date, station_id, line_id, COUNT(*)
FROM station_daily_statistics
GROUP BY date, station_id, line_id
HAVING COUNT(*) > 1;
-- → 0 rows (correct)
```
